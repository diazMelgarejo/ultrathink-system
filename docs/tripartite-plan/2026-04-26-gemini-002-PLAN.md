# Plan: Hardware-Enforced Model Routing & Shared Intelligence

## Objective
Establish a hard machine-enforced policy for model-to-hardware routing across the three-repo architecture to prevent hardware damage on Mac (overload) and resource under-utilization on Windows.

## Model Affinity Definitions
| Model ID | Affinity | Reason |
|----------|----------|--------|
| `gemma-4-26b-a4b-it` | **Windows-Only** | Exceeds Mac VRAM/cooling safety limits. |
| `qwen3.5-27b-claude-4.6-opus-reasoning-distilled-v2` | **Windows-Only** | Exceeds Mac VRAM/cooling safety limits. |
| `gemma-4-e4b-it` | **Mac-Only** | Optimized for Apple Silicon; under-utilizes Windows GPU. |
| `qwen3.5-9b-mlx` | **Mac-Only** | MLX weights required; native to Apple Silicon. |

## 1. Enforcement Strategy (The "Rama" Way)
Unlike the Codex plan which focuses on filtering discovery, my plan focuses on **runtime enforcement** at the orchestrator level. Even if a user manually adds a model or discovery fails, the system will block invalid assignments.

### A. Perpetua-Tools (L2 - Adapter/Middleware)
- **`hardware/SKILL.md`**: Update the "Constraint" column in the Role Matrix to explicitly state `NEVER_MAC` or `NEVER_WIN` for the models above.
- **`orchestrator/alphaclaw_manager.py`**: Add a `validate_routing_affinity(model_id, platform)` function. Call this during the `--resolve` sequence. If an affinity violation is detected, raise a standard AlphaClaw error message: `[alphaclaw] Fatal: Hardware Mismatch. Model {model_id} cannot run on {platform}.`
- **`agent_launcher.py`**: Inject the same validation into the probe loop.

### B. orama-system (L3 - Intelligence/Orchestration)
- **`bin/config/agent_registry.json`**: Add an `affinity` key to the `openclaw_agents` and `autoresearch_agents` definitions.
- **`api_server.py`**: Add a pre-flight check that queries the PT `alphaclaw_manager` for affinity clearance before spawning an agent.

## 2. Documentation & Shared Intelligence
- **`AGENT_RESUME.md` (Both Repos)**: Add a "Hardware Safety & Model Affinity" section to the "Current Status" to ensure the next agent is aware of these hard boundaries.
- **`LESSONS.md` (Both Repos)**: Document the "2026-04-26 Hardware-Model Alignment" lesson, specifically detailing the "fry Mac hardware" and "under-utilize Windows" scenarios as high-risk failure modes.

## 3. Implementation Course of Action
1.  **Phase 1: Knowledge Update**: Edit `SKILL.md` and `MODEL_HARDWARE_MATRIX.md` to define the policy.
2.  **Phase 2: L2 Enforcement**: Refactor PT's manager and launcher to validate these rules at runtime.
3.  **Phase 3: L3 Integration**: Update Orama's registry and API server to consume the PT safety signals.
4.  **Phase 4: Forensics & Crystallization**: Update Resume and Lessons files.

### Why my idea is good:
The "Ghost Orchestrator" must be a **Safe Orchestrator**. Relying on a discovery script (`discover.py`) to filter models is a "soft" guard. My plan implements a "hard" guard in the L2/L3 interface. By making PT's manager authoritative for hardware affinity, we create a single point of failure (or safety) that is easier to audit and harder to bypass.

## Recommendation
I suggest we **combine the plans**:
1. Use Codex's **Discovery Filtering** to keep the config files clean.
2. Use Gemini's **Runtime Enforcement** to provide the safety "kill-switch".
3. Apply Gemini's **Resume & Lessons** updates for long-term intelligence sharing.
